# 需求规格说明书（SRS）

**项目名称**：基于知识图谱与大语言模型的主动问诊智能Agent研究
**版本**：V1.0 (MVP 最小可行性产品版)

## 1. 项目概述

### 1.1 项目背景

传统医疗大模型大多采用“被动问答（Reactive QA）”范式，患者输入什么，模型回答什么。然而，真实的临床问诊是一个“主动追问（Proactive Consultation）”的过程，医生需要通过不断询问隐性症状来缩小疾病假设域。本项目旨在结合知识图谱的确定性知识（GraphRAG）和大语言模型（LLM）的逻辑推理能力，构建一个能主动追问、自我规划的智能医疗 Agent。

### 1.2 核心目标

打造一个能在网页端运行的对话交互系统：当患者输入初始不适症状时，Agent 能够结合底层医学图谱进行推理，通过多轮追问获取完整的患者状态，并最终输出疾病诊断与建议。

------



## 2. 系统技术栈与架构设计

为严格控制 MVP 范围并聚焦算法逻辑，本系统技术栈如下：

- 
- **前端与交互层**：Streamlit (纯 Python 构建的 Web 对话 UI 框架，快速实现类似 ChatGPT 的多轮对话界面)。
- **核心大脑 (LLM)**：DeepSeek API (提供强大的指令遵循与思维链推理能力，通过官方 URL 和 API Key 调用，免除本地算力困扰)。
- **知识存储层**：Neo4j (图数据库，存储“疾病-症状”等实体关系)。
- **Agent 与 RAG 框架**：纯 Python 脚本 + 提示词工程 (Prompt Engineering) 开发核心机制，或轻量级引入 LangChain 用于大模型和图谱的胶水层连接。

## 3. 功能需求说明 (MVP)

### 功能模块 1：基于 Streamlit 的多轮对话交互界面

- 
- **F1.1 初始状态输入**：提供类似聊天框的输入栏，允许用户用自然语言输入初始症状（例：“我最近三天老是咳嗽”）。
- **F1.2 对话历史保持**：在界面中完整展示历史对话记录（User 与 Agent 的多轮交互），利用 Streamlit 的 st.session_state 维护当前问诊的上下文。

### 功能模块 2：知识图谱初始化与预留扩展层（你的需求 #2）

- 
- **F2.1 结构化数据快速导入 (MVP核心)**：提供 Python 脚本，一键读取 MuZhi/Dxy 开源数据集中的 CSV/JSON 格式“症状-疾病”关系网络，通过 neo4j Python 驱动写入图数据库中。
- **F2.2 非结构化建图扩展接口 (未来扩展)**：预留一个模块接口 build_kg_from_text(text: str)。当前版本可以直接 pass，但架构上允许后续引入 LLM 的 NER（命名实体识别）和 RE（关系抽取）功能，处理自由文本更新到 Neo4j。

### 功能模块 3：患者意图与实体抽取 (NLU)

- 
- **F3.1 症状实体抽取**：用户输入自然语言后，通过调用 DeepSeek API 将其转化为结构化症状实体列表（如：“今天头有些疼还有点恶心” -> [头痛, 恶心]）。
- **F3.2 意图识别**：识别用户当前是“陈述症状”、“回答 Agent 的追问(Yes/No/不知道)”，还是“结束问诊”。

### 功能模块 4：基于 GraphRAG 的主动规划问诊 (核心创新点！)

- 
- **F4.1 图谱子图检索**：根据当前抽取出的症状列表，查询 Neo4j 数据库。寻找同时包含这些症状的候选疾病（如疾病A、疾病B），并提取这些**候选疾病周边的未确认症状节点**，形成“上下文子图（Context Subgraph）”。
- **F4.2 大模型 Agent 决策与追问生成 (ReAct/CoT)**：
  - 
  - 将【当前患者已有症状】+【图谱检索到的子图信息】拼接注入到 DeepSeek 的 Prompt 中。
  - 利用 DeepSeek 的思维链（CoT），让 Agent 判断：“当前证据是否足以确诊？”
  - 如果**不足以确诊**：Agent 在检索到的周边症状中，挑选最具有区分度的一个症状，生成自然语言追问（例如：“那请问您有没有感觉到胸闷气短？”）。
- **F4.3 状态流转控制**：设定最大追问轮数（例如 5 轮）。防止 Agent 陷入死循环死板追问。

### 功能模块 5：最终诊断与总结建议

- 
- **F5.1 阈值触发与诊断**：当确诊置信度足够高，或者追问达到最大轮数时，Agent 停止追问。
- **F5.2 综合建议生成**：将多轮对话收集到的所有显性与隐性症状交给 DeepSeek，结合底层图谱给出最终可能的疾病结论以及初步就医/用药建议。

## 4. 数据集与评估策略约束

- 
- **底层图谱构建与测试来源**：采用 **MuZhi (拇指) / Dxy (丁香园) 症状-疾病对话数据集**。
- **系统成功验证标准**：
  1. 
  2. **主动性命中率 (Proactive Accuracy)**：Agent 在面对测试集中的残缺症状输入时，追问的下一个症状，是否属于该数据集该病例的“真实隐性症状”？
  3. **诊断准确率 (Diagnostic Accuracy)**：经历 N 轮追问后，最终输出的 Top-3 诊断结论是否命中真实疾病？

## 5. 核心业务流程流转图 (Agent Logic Flow)

```
User Input (症状描述)
       ↓[模块3: DeepSeek 实体抽取] -> 得到症状实体集合 S
       ↓[模块4: Neo4j 图查询] -> 检索得到包含 S 的候选疾病，以及区分这些疾病的关键未确认症状集 C
       ↓
[模块4: GraphRAG + DeepSeek Planning]
Prompt: "患者现有症状: S。图谱关联提示: C。请判断是否能确诊？若不能，请选择一个最有价值的症状进行追问。"
       ↓
   < Decision Branch >
  /                   \
(需补充信息)        (可确诊 / 达到最大轮数)
  /                       \
Agent生成追问句子[模块5: 结合所有S和疾病进行最终诊断]
(如：你有流鼻涕吗)                  ↓
  ↓                           Agent输出就医建议
User 回答 (是/否)                   ↓
  ↓                                End
更新症状集合 S
(循环回图查询...)
```
