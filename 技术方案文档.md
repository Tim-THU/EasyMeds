# 《基于知识图谱与大模型的主动问诊 Agent 技术实现方案》（MVP版）

## 1. 架构总览 (Architecture Overview)

本系统采用端云结合的轻量化架构。前端不设复杂的 Web 框架，由纯 Python 的 Streamlit 驱动；系统大脑由 LangGraph 接管对话状态流转，通过统一的 OpenAI SDK 路由调用 DeepSeek-V3 与 DeepSeek-R1；知识存储依托免本地部署的 Neo4j AuraDB 云端图数据库。

### 1.1 核心技术栈确认

- **交互层**：Streamlit (V1.30+，利用 st.chat_message 渲染多轮对话)。
- **状态编排层**：LangGraph (构建图结构的循环状态机，控制最高 5 轮的主动追问)。
- **大语言模型**：DeepSeek API
  - deepseek-chat (V3模型)：用于前置的意图识别与症状实体抽取（速度快、成本低）。
  - deepseek-reasoner (R1模型)：用于带有思维链 (CoT) 的复杂图谱子图推理与问诊策略生成。
- **知识存储**：Neo4j AuraDB (免费云端版，提供基于 Cypher 的 GraphRAG 检索)。
- **数据初始化**：通过 requests 调用 GitHub REST API（或 Raw URL）动态拉取 MuZhi/Dxy 数据集 [1]。

### 1.2 系统架构与数据流向图 (Mermaid)

```
graph TD
    subgraph Frontend [Streamlit 前端交互层]
        UI[患者对话界面 UI]
        Session[Session State 维持]
    end

    subgraph LangGraph_Agent[LangGraph 主动问诊状态机]
        State(State: 维护已有症状, 候选疾病, 对话轮数)
        Node_NLU[节点1: NLU 与实体抽取]
        Node_Graph[节点2: 图谱关联检索 - GraphRAG]
        Node_Reason[节点3: R1 追问与决策规划]
        
        State --> Node_NLU --> Node_Graph --> Node_Reason
        Node_Reason -- 追问轮数<5 且 未确诊 --> UI
        Node_Reason -- 达到阈值 --> Node_Diag[节点4: 最终诊断建议]
    end

    subgraph Cloud_Services [云端接口服务]
        DS_Chat(DeepSeek-Chat API\n意图/实体抽取)
        DS_Reasoner(DeepSeek-Reasoner API\n思维链推理)
        Neo4j[(Neo4j AuraDB\n疾病-症状图谱)]
        Github(GitHub Raw API\n动态拉取数据集)
    end

    UI -->|自然语言输入| Node_NLU
    Node_NLU <-->|API 调用| DS_Chat
    Node_Graph <-->|Cypher 查询| Neo4j
    Node_Reason <-->|带 Context 提示词| DS_Reasoner
    Github -->|初始化图谱| Neo4j
```

## 2. 模块级技术实现细节

### 2.1 模块一：数据拉取与图谱初始化 (GitHub -> Neo4j)

**实现逻辑**：系统首次启动时（或提供一个隐藏的初始化按钮），无需用户本地下载 CSV/JSON，直接通过 Python 的 requests 库访问 GitHub Raw Content URL 动态拉取开源数据集（例如 Dxy 数据集）。

- 
- **API 规范**：使用 GitHub raw 链接格式：https://raw.githubusercontent.com/{owner}/{repo}/main/{path_to_csv} [2]。
- **图谱构建 (Cypher 建模)**：
  数据拉取解析为 Pandas DataFrame 后，通过官方 neo4j Python driver 写入 AuraDB。
  MVP 最小化图模式设定为两种节点和一种关系：
  (s:Symptom {name: "头痛"})-[:INDICATES {weight: 0.8}]->(d:Disease {name: "感冒"})

### 2.2 模块二：NLU 与实体抽取层 (DeepSeek-Chat)

**实现逻辑**：患者的自然语言往往是模糊的（如“我脑袋快炸了”）。我们将患者的最新输入交给 deepseek-chat。

- 
- **Prompt 约束**：要求模型以严格的 JSON 格式输出，提取出医学标准症状词（Mapping）。
- **输出示例**：{"extracted_symptoms": ["头痛", "剧烈"]}。将提取出的症状追加到 LangGraph 的全局 State 变量 confirmed_symptoms 中。

### 2.3 模块三：基于 Neo4j 的 GraphRAG 检索层

**实现逻辑**：Agent 获取到最新的 confirmed_symptoms 列表后，自动生成并执行 Cypher 查询。

- 
- **检索目标 1 (候选疾病)**：查找同时关联这些已知症状的所有 Disease 节点。
- **检索目标 2 (周边隐性症状)**：基于找出的候选疾病，反向查出这些疾病**还关联了哪些患者尚未提及的症状**（Unconfirmed Symptoms）。将这两块信息合并为 Graph_Context（图谱上下文）。

### 2.4 模块四：基于 DeepSeek-Reasoner 的 Agent 决策与规划（核心）

**实现逻辑**：这是体现“主动问诊”的灵魂模块。将上一步的 Graph_Context 结合当前病史，注入给带有强大思维链的 deepseek-reasoner。

- 
- **API 调用特性**：在 2026 年的 OpenAI 兼容 SDK 调用中，设置 model="deepseek-reasoner"，模型会返回 <think> 标签内的思考过程 [8]。
- **Prompt 策略设计**：
  - 
  - *System Prompt*: "你是一个主动问诊专家。已知患者当前症状：[已有症状]。基于底层图谱提示，候选疾病有：[疾病A、B]。这些疾病区分的关键症状是：[未确认症状 X、Y]。"
  - *任务约束*: "请你先思考是否能唯一确诊？如果不能，请挑选[未确认症状 X、Y]中最具区分度的一个，用温和的语气向患者发起追问。如果能确诊，输出最终诊断协议。"
- **流转控制**：解析模型的响应内容，如果属于“追问”，则将问题抛回给 Streamlit 界面等待患者回答；如果属于“确诊”，则流转至诊断节点。

### 2.5 模块五：LangGraph 循环状态机与前端 Streamlit 集成

**实现逻辑**：传统的循环写 while 容易失控，利用 LangGraph 的 StateGraph 来精准控制。

- **State 定义**：

  

  ```
  class ConsultationState(TypedDict):
      messages: Annotated[list, add_messages]
      confirmed_symptoms: list
      turn_count: int  # 严格控制轮数
  ```

- **条件边 (Conditional Edges)**：在 Agent 推理节点后接入一个判断逻辑。如果 turn_count >= 5 或 Agent 输出标志着“已确诊”，则强制进入 Final_Diagnosis 节点；否则回到 User_Input 节点。

- **前后端桥接**：Streamlit 端的 st.session_state["thread_id"] 绑定 LangGraph 的会话持久化内存 (MemorySaver)，确保用户在网页刷新时依然处于同一个问诊流中。

## 3. 环境变量与部署配置规范

为了保障系统的安全性与可移植性（不在代码中硬编码任何密码），部署前需在根目录配置 .env 文件：

```
# 核心大模型 API 配置 (DeepSeek)
DEEPSEEK_API_KEY="sk-xxxxxxxxxxxxxxxxxxx"
DEEPSEEK_BASE_URL="https://api.deepseek.com" # 使用OpenAI兼容SDK时的标准BaseURL

# 知识图谱云端库配置 (Neo4j AuraDB)
NEO4J_URI="neo4j+s://xxxxxxxx.databases.neo4j.io"
NEO4J_USERNAME="neo4j"
NEO4J_PASSWORD="your-cloud-password"

# GitHub 数据动态获取 (若拉取私有库需Token，公有库可留空)
GITHUB_TOKEN=""
```

## 4. MVP 开发实施路径规划 (实施建议)

作为架构师，我建议你按照以下 3 个 Sprint（冲刺周期）来逐步落地代码，确保不会在不重要的细节上卡住：

- **Sprint 1 (图谱基建阶段)**：跑通 Python 脚本，用 requests 从 GitHub 下载指定的 CSV，使用 Cypher 将症状与疾病的关联写入你的 AuraDB 实例。验证可以在 AuraDB 的控制台中通过语句查出关系。
- **Sprint 2 (核心大脑阶段)**：不写 UI 界面，直接在终端里用命令行跑 LangGraph 逻辑。接入 deepseek-chat 提取你输入的症状，查 Neo4j，并用 deepseek-reasoner 生成追问。跑通这一个死循环。
- **Sprint 3 (前端组装阶段)**：引入 Streamlit 编写 app.py，将 Sprint 2 中的终端输入输出重定向到 Streamlit 的 Web 聊天框中。完成系统交付！
